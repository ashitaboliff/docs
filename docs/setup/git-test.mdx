# Gitを練習しよう
開発環境を作成している途中ですが、Gitの基本的な概念とあしたぼホームページ開発でのGit/GitHubを使った開発フローについて説明していきます。

## 1. コミット/プッシュ
Gitではコードの変更を**コミット**という単位で管理します。ライザップすぎる言い回しですが普通に「変更を保存する」だと考えてください。  
コミットには変更内容を説明するメッセージを付ける義務があり、このメッセージは他者への情報共有にも、自分自身の作業記録としても役に立ちます。

コミットなど、変更をリモートリポジトリ（GitHubなど）に反映させることを**プッシュ**といいます。  
プッシュすることで、はじめて他の開発者と変更内容を共有できるようになります。コミットはあくまで自分のPC上のみでの話です。

## 2. プル
Gitでは、他の開発者が行った変更を自分のローカルリポジトリに取り込むことを**プル**といいます。
プルすることで、他の開発者が行った変更を自分のローカルリポジトリに反映させることができます。  

:::warning
チーム開発では、こまめにプルを行わないと、他の開発者と自分の変更が**競合**(コンフリクト)してしまうことがあります。  
コンフリクトを解消するのはぼちぼちめんどくさくてそこそこ難しいので出来る限り起こさないようにプルはこまめに行いましょう。
:::

## 3. ブランチ
Gitでは**ブランチ**という概念があります。日本語で「枝」です。  
ブランチはコードの変更履歴を管理するための仕組みで、複数の開発者が同時に異なる機能を開発したり、バグ修正を行ったりする際に役立ちます。
あしたぼホームページ開発ではこのブランチを活用して、各自が独立して作業できるように、かつ本番環境に影響を与えないようにしています。

図2は実際のあしたぼホームページ フロントエンドのブランチ構成です。ピンク色の線で表されているものが`master`ないしは`origin/master`で、本番環境に対応するブランチです。  
青色の線で示されているものが開発環境の中心となる`dev`ブランチで、各自が作業するブランチはこの`dev`ブランチから分岐して作られています。  
実際に`feat/react-doctor`や`feat/ui-refactor`が開発ブランチの名前で、ブランチの名前は`feature/〇〇`や`fix/〇〇`など、変更内容がわかるような名前にしています。  

:::note
これらのブランチの変更を、後述する**マージ/プルリクエスト**の機能で`dev`ブランチに吸収させ、さらにリリースのタイミングで`dev`ブランチから`master`ブランチにマージし、本番環境に反映される、という流れになっています。
:::

:::tip 
ブランチの名前の先や後ろについている`origin/`はリモートリポジトリのブランチであることを示しています。まあGitHub上にアップロード(プッシュ)されたブランチと思ってください。
:::

## 4. マージ/プルリクエスト
Gitでは、あるブランチの変更を別のブランチに取り込むことを**マージ**といいます。
マージは、変更内容を別のブランチに統合するための操作で、複数の開発者が同時に開発した異なる機能を一つのブランチにまとめる際に使用されます。

あしたぼホームページ開発では、GitHubの**プルリクエスト**機能を使用して、マージのプロセスを管理しています。  
プルリクエストは、あるブランチの変更を別のブランチにマージする前に、変更内容をレビューするための機能です。プルリクエストを作成すると、他の開発者が変更内容を確認し、コメントを残すことができます。変更内容に問題がなければ、プルリクエストを承認してマージすることができます。

## 5. Git/GitHubを使った開発フロー
長々とGitの基本的な概念を説明してきましたが、実際の開発フローは以下のようになります。

:::info
あしたぼホームページ開発では`Git Graph`というVSCodeの拡張機能の利用を推奨してます！本当はGitはCLIで操作するものですが普通にわかりづらいと思うのでこっち使ってください
:::

<StepList>
1.  ### ブランチの作成
    `dev`ブランチから新しいブランチ(`feat/○○`)を作成します！    
    `Git Graph`を入れているとVSCode上の「ソース管理」から該当リポジトリに対して「ブランチ」→「ブランチの作成」で簡単にできます。
    <Figure
      src="https://docs-cdn.ashitabo.net/setup/4-2.jpg"
      alt="Git Graphでのブランチ作成のスクショ"
      width={600}
      caption="Git Graphでのブランチ作成のスクショ"
    />
    コマンドラインからも以下のコマンドでできます。
    ```bash
    git checkout -b feat/○○ dev
    ```
    これで`dev`ブランチから`feat/○○`ブランチが作成され、さらにそのブランチに切り替わります。
2.  ### コードの変更とコミット/プッシュ
    `feat/○○`ブランチ上でコードの変更を行い、こまめにコミット/プッシュをします。    
    キリのいいところ(一つ小さな機能が出来た、作業をいったん中断するなど)でこまめにコミットをしましょう。

    :::warning
    コミットメッセージは僕はかなり適当にしていましたが、普通に他人にわかりやすいようにしましょう。  
    機能追加なら「feat: `/admin/test`に○○を追加」、バグ修正なら「fix: `/admin/test`の○○のバグを修正」みたいな感じで、変更内容が簡潔にわかるようにするのが理想です。  
    GitHub Copilotを入れておくとコミットメッセージの自動生成機能もあります！微妙だけど！
    :::

    CLIでのコミット/プッシュは以下のコマンドでできます。
    ```bash
    git add <対象ファイル>
    git commit -m "コミットメッセージ"
    git push origin feat/○○
    ```

3. ### プルリクエストの作成
    変更が完了したら、GitHub上で`feat/○○`ブランチから`dev`ブランチへのプルリクエストを作成しましょう。  
    完全にタスクが完了してからプルリクエストを作成するのもいいですが、途中段階で作成しておいて、変更内容を他の開発者にレビューしてもらうのもおすすめです。なんか下書き機能とかあるし
    <Figure
      src="https://docs-cdn.ashitabo.net/setup/4-3.jpg"
      alt="GitHubでのプルリクエスト作成のスクショ"
      width={600}
      caption="GitHubでのプルリクエスト作成のスクショ"
    />

    :::warning
    プルリクエストを作成する際は、`feat/○○`ブランチから`dev`ブランチへのマージとなるので`base`は`dev`、`compare`は`feat/○○`を選択してください。  
    まあ間違えても修正できるから大丈夫だけどねー
    :::
4. ### レビュー
    ここまで出来たら他の開発者にレビューしてもらいましょう！  
    プルリクエストを作成すると、他の開発者が変更内容を確認し、コメントを残すことができます。変更内容に問題がなければ、プルリクエストを承認してマージすることができます。  
    :::tip
    Discordの`#プルリク部屋`でレビュアーをメンションしてレビューを依頼しとくといいです。  
    GitHubの通知、微妙だからねー
    :::
</StepList>

これがあしたぼホームページ開発の基本的なGit/GitHubを使った開発フローになります。この開発の積み重ねでいいタイミングで`dev`ブランチから`master`ブランチにマージしてリリースする、という流れになります。

## 6. 注意点
Git/GitHubを使った開発フローでは、正直そこそこ複雑なソフトウェアなので、いろんなトラブルが起こる可能性があります。以下の3点を特に注意して開発を進めましょう。
- **こまめにプルする**  
    他の開発者と自分の変更が競合(コンフリクト)してしまうことを防ぐために、こまめに`dev`ブランチに対して、その変更を同期するプルを行いましょう。
- **コミットメッセージはわかりやすく**  
    コミットメッセージは他人にわかりやすいようにしましょう。変更内容が簡潔にわかるようにするのが理想です。
- **差分をよく確認する**
    プルリクエストを作成する前に、変更内容の差分をよく確認しましょう。意図しない変更が含まれていないか、コードの品質が保たれているかなどを確認することが重要です。

んまあこうして注意していても全然トラブルは起こります。普通に一人でやっててもめんどくさいことなるもん。まあ何とかなるのでのびのびやりましょう。

## 7. 実際にやってみる
ここまででGit/GitHubの基本的な概念とあしたぼホームページ開発でのGit/GitHubを使った開発フローについて説明してきました。まあ普通にこれらの操作はとっても難しいと思います。
慣れるまでとても時間がかかるし、たくさん失敗すると思います(ぼくもそうだったし)。なのでまずはルートの`ashitabo`リポジトリを使って実践してみましょう！
<StepList>
1. ### `ashitabo`リポジトリの`main`ブランチから新しいブランチを切る

    `feat/○○-test`というブランチを`main`から作りましょう！
    ○○の部分には自分のユーザ名とかなんか識別できるものを入れてください。  
    このブランチ上でコードの変更を行いましょう。
    :::tip
    `ashitabo`リポジトリはあしたぼホームページのコードが入っているリポジトリではなく、あしたぼホームページ開発のためのドキュメントやスクリプトが入っているリポジトリです。
    なので、ここで行う変更はあしたぼホームページのコードには影響しません。  
    あとまともにブランチ使ってないし今回は`main`ブランチでOK！
    :::
2. ### 少し変更してみる

    例えば、
    - `○○-unko.txt`というテキストファイルを作ってみる。
    - `○○-unko.md`というマークダウンファイルを作ってうんこの絵文字で埋めてみる。
    - `○○-unko.js`というJavaScriptファイルを作ってconsole.logでうんこと出力してみる。
    
    など、好きにしてくれて構いません。
3.  ### コミット/プッシュをする

    変更内容をコミットして、GitHubにプッシュしてみましょう。その際に適切に変更内容を記述したコミットメッセージを書いてみましょう。
4.  ### プルリクエストを行う

    `feat/○○-test`ブランチから`main`ブランチにマージをするためのプルリクエストをGitHub上で出しましょう。  
    右側に`Reviewers`という項目があるので、そこにwatabeggをレビュアーを追加してみましょう。
5.  ### レビュアーにレビューしてもらう

    Discordの`#プルリク部屋`でレビュアーをメンションしてレビューを依頼してみましょう。  
    レビュアーからコメントが来ると思うので、適切に返信してみたり無視したりしてみましょう。
6.  ### プルリクエストをマージする

    レビュアーから承認されたら、プルリクエストをマージしてみましょう。
    これで`main`ブランチに変更が反映されました。
</StepList>

上記の操作は`ashitabo`リポジトリというあしたぼホームページ開発のためのサポートファイルなのであしたぼホームページの本番環境には影響しません。**失敗しても大丈夫です！**  
この一連の流れを実際に体験してみることで、Git/GitHubを使った開発フローに慣れることができます。
最初は戸惑うかもしれませんが、繰り返し行うことで自然と理解できるようになります。頑張ってください！

:::warning
またこのときに作ったファイルとか変更はいつか僕が勝手に消しときます。消さないかもしれません。
:::

次に！このGit/GitHubの知識をフル活用して、実際にあしたぼホームページの開発環境を立ち上げてみましょう。
